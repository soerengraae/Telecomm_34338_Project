{\bfseries{Positioning tooltips and popovers is difficult. Popper is here to help!}}

Given an element, such as a button, and a tooltip element describing it, Popper will automatically put the tooltip in the right place near the button.

It will position {\itshape any} UI element that "{}pops out"{} from the flow of your document and floats near a target element. The most common example is a tooltip, but it also includes popovers, drop-\/downs, and more. All of these can be generically described as a "{}popper"{} element.\hypertarget{README.md_autotoc_md3037}{}\doxysubsubsection{\texorpdfstring{Demo}{Demo}}\label{README.md_autotoc_md3037}

\begin{DoxyItemize}
\item {\bfseries{Clipping and overflow issues}}\+: Pure CSS poppers will not be prevented from overflowing clipping boundaries, such as the viewport. It will get partially cut off or overflows if it\textquotesingle{}s near the edge since there is no dynamic positioning logic. When using Popper, your popper will always be positioned in the right place without needing manual adjustments.
\item {\bfseries{No flipping}}\+: CSS poppers will not flip to a different placement to fit better in view if necessary. While you can manually adjust for the main axis overflow, this feature cannot be achieved via CSS alone. Popper automatically flips the tooltip to make it fit in view as best as possible for the user.
\item {\bfseries{No virtual positioning}}\+: CSS poppers cannot follow the mouse cursor or be used as a context menu. Popper allows you to position your tooltip relative to any coordinates you desire.
\item {\bfseries{Slower development cycle}}\+: When pure CSS is used to position popper elements, the lack of dynamic positioning means they must be carefully placed to consider overflow on all screen sizes. In reusable component libraries, this means a developer can\textquotesingle{}t just add the component anywhere on the page, because these issues need to be considered and adjusted for every time. With Popper, you can place your elements anywhere and they will be positioned correctly, without needing to consider different screen sizes, layouts, etc. This massively speeds up development time because this work is automatically offloaded to Popper.
\item {\bfseries{Lack of extensibility}}\+: CSS poppers cannot be easily extended to fit any arbitrary use case you may need to adjust for. Popper is built with extensibility in mind.
\end{DoxyItemize}\hypertarget{README.md_autotoc_md3040}{}\doxysubsubsection{\texorpdfstring{Why Popper?}{Why Popper?}}\label{README.md_autotoc_md3040}
With the CSS drawbacks out of the way, we now move on to Popper in the Java\+Script space itself.

Naive Java\+Script tooltip implementations usually have the following problems\+:


\begin{DoxyItemize}
\item {\bfseries{Scrolling containers}}\+: They don\textquotesingle{}t ensure the tooltip stays with the reference element while scrolling when inside any number of scrolling containers.
\item {\bfseries{DOM context}}\+: They often require the tooltip move outside of its original DOM context because they don\textquotesingle{}t handle {\ttfamily offset\+Parent} contexts.
\item {\bfseries{Compatibility}}\+: Popper handles an incredible number of edge cases regarding different browsers and environments (mobile viewports, RTL, scrollbars enabled or disabled, etc.). Popper is a popular and well-\/maintained library, so you can be confident positioning will work for your users on any device.
\item {\bfseries{Configurability}}\+: They often lack advanced configurability to suit any possible use case.
\item {\bfseries{Size}}\+: They are usually relatively large in size, or require an ancient j\+Query dependency.
\item {\bfseries{Performance}}\+: They often have runtime performance issues and update the tooltip position too slowly.
\end{DoxyItemize}

{\bfseries{Popper solves all of these key problems in an elegant, performant manner.}} It is a lightweight \texorpdfstring{$\sim$}{\string~}3 kB library that aims to provide a reliable and extensible positioning engine you can use to ensure all your popper elements are positioned in the right place.

When you start writing your own popper implementation, you\textquotesingle{}ll quickly run into all of the problems mentioned above. These widgets are incredibly common in our UIs; we\textquotesingle{}ve done the hard work figuring this out so you don\textquotesingle{}t need to spend hours fixing and handling numerous edge cases that we already ran into while building the library!

Popper is used in popular libraries like Bootstrap, Foundation, Material UI, and more. It\textquotesingle{}s likely you\textquotesingle{}ve already used popper elements on the web positioned by Popper at some point in the past few years.

Since we write UIs using powerful abstraction libraries such as React or Angular nowadays, you\textquotesingle{}ll also be glad to know Popper can fully integrate with them and be a good citizen together with your other components. Popper is distributed in 3 different versions, in 3 different file formats.

The 3 file formats are\+:


\begin{DoxyItemize}
\item {\ttfamily esm} (works with {\ttfamily import} syntax â€” {\bfseries{recommended}})
\item {\ttfamily umd} (works with {\ttfamily \texorpdfstring{$<$}{<}script\texorpdfstring{$>$}{>}} tags or Require\+JS)
\item {\ttfamily cjs} (works with {\ttfamily require()} syntax)
\end{DoxyItemize}

There are two different {\ttfamily esm} builds, one for bundler consumers (e.\+g. webpack, Rollup, etc..), which is located under {\ttfamily /lib}, and one for browsers with native support for ES Modules, under {\ttfamily /dist/esm}. The only difference within the two, is that the browser-\/compatible version doesn\textquotesingle{}t make use of {\ttfamily process.\+env.\+NODE\+\_\+\+ENV} to run development checks.

The 3 versions are\+:


\begin{DoxyItemize}
\item {\ttfamily popper}\+: includes all the modifiers (features) in one file ({\bfseries{default}});
\item {\ttfamily popper-\/lite}\+: includes only the minimum amount of modifiers to provide the basic functionality;
\item {\ttfamily popper-\/base}\+: doesn\textquotesingle{}t include any modifier, you must import them separately;
\end{DoxyItemize}
